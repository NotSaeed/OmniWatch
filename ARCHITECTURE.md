# OmniWatch Zero-Knowledge Architecture

## 1. Executive Summary
OmniWatch is an exploratory cyber-physical testbed and verifiable evaluator for out-of-band OT triage. The architecture serves a dual mandate: it functions as a rigorous academic benchmark evaluating the latency overheads of Zero-Knowledge Virtual Machines (zkVMs) at the edge, while simultaneously providing a highly deployable, compliance-ready enterprise security product.

It processes Modbus TCP payloads through a deterministic invariant library, utilizing a RISC-V STARK prover to provide computationally sound verification (~96-bit conjectured security) of policy evaluation over observed telemetry (ensuring Computational Integrity). This is gated alongside a FIDO2-backed ECDSA cryptographic "Proof of Oversight" for human authorization. Following verification, a localized Small Language Model (Phi-3-Mini) acts strictly as a Retrieval-Augmented Generation (RAG) translation engine to explain invariant violations using static facility documentation, constrained by strict visual source grounding to prevent automation bias.

## 2. Bounded Threat Model and Central Limitations
To ensure rigorous academic validity for exhibition and enterprise defensibility for commercial evaluation, the architecture explicitly defines its operational boundaries:
• **The Latency vs. Automation Benchmark:** The system explicitly acknowledges that RISC Zero STARK generation (9–17 seconds) combined with 4-bit Phi-3-Mini CPU inference (10–25 tokens/second) creates a total end-to-end operator triage latency of 15–32 seconds. Furthermore, a one-time Initial WASM Compilation Penalty (parsing a 10–20MB verifier bundle on a cold browser cache) delays initial dashboard readiness. This validates the project's core research hypothesis: Because current verifiable computation and local generative AI are too slow for real-time OT mitigation, cryptographically signed Human-in-the-Loop authorization remains the superior operational security control for post-detection governance.
• **Edge-Node Trust Boundary & Pre-Proof Tampering (Central Limitation):** The trust boundary terminates at the Raspberry Pi 4 edge node. The system lacks TPM 2.0 remote attestation and Secure Boot. Because the Pi 4 serializes the JSON telemetry into a bincode payload prior to zkVM evaluation, the Pi 4 OS represents the primary attack vector for pre-proof data manipulation. Consequently, the architecture achieves strict Computational Integrity but lacks hardware-rooted Sensing Integrity, functioning as a verifiable evaluator of supplied data rather than an infallible monitor of physical reality.
• **Transparent Cryptography & Proof Size:** To maintain uncompromised cryptographic transparency and a setup-free environment, the pipeline explicitly pins the receipt type to succinct STARK receipts, rejecting optional SNARK wrappers. Raw STARK proofs incur larger sizes (~217–250 KB), but this overhead is negligible on a local high-speed network.
• **Physical Layer Fidelity (Pedagogical Simulation):** The physical diorama utilizes an ESP32 microcontroller operating Modbus TCP over Wi-Fi. Because NCA OTCC-1:2022 mandates strict restriction and segmentation of wireless technologies in production OT/ICS, this setup functions strictly as a pedagogical simulation rather than a representative deployment architecture. The exhibition incorporates a latency comparison baseline to quantify the deviation between standard deterministic RS-485 (<= 1ms cycle time, 0 ms CSMA/CA overhead) and the ESP32 Wi-Fi simulation (~10–50ms variable cycle time).
• **Static Invariant Recompilation (v1 Tuning Hook):** To ensure immutable cryptographic tracking of the exact policy version evaluated, the deterministic invariants are hardcoded into the Rust Guest Code. Adding new threat intelligence rules requires a full recompile of the zkVM guest program. This is accepted as a v1 trade-off, sacrificing dynamic hot-reloading in favor of absolute state immutability.
• **Cryptographic Reproducibility:** To ensure cryptographic reproducibility against breaking API changes in the fast-moving zero-knowledge ecosystem, the exact risc0-zkvm crate version is strictly pinned and documented within the threat model appendix.

## 3. Hardware Partitioning and Physical Simulation
The infrastructure utilizes strict physical service isolation to prevent computational bottlenecks.
• **The OT Target (Victim):** An ESP32 Type C WiFi/Bluetooth Module operating a Modbus TCP server, functionally simulating a localized SCADA network segment physically wired to 5V fluid pumps and LED status indicators.
• **The Attacker Simulation (Red Team):** A Raspberry Pi Zero 2W paired with a TP-Link TL-WN722N Network Adapter, executing the localized injection of unauthorized Modbus FC 05 and FC 16 payloads.
• **Live Demonstration Fallback:** To mitigate the extreme 2.4GHz Wi-Fi spectrum saturation inherent to exhibition environments, the pipeline includes a Deterministic Replay Fallback mode, allowing the ingest of pre-recorded PCAP captures to guarantee presentation continuity. To entirely eliminate network-based truncation attacks, the FastAPI server is granted only SELECT and INSERT privileges. Truncating the SQLite Spent-Receipt Registry to allow PCAP replays is defined as an Out-of-Band (OOB), CLI-only operation executed directly on the Command Node OS, an accepted procedural bypass explicitly scoped to pedagogical contexts.
• **The Network Sensor (Blue Team Edge Node):** A passive Raspberry Pi 4 Model B (with 128GB SanDisk Ultra Micro SD) operating out-of-band via a network TAP. It executes Zeek, augmented with the CISA ICSNPP Modbus package to accurately capture originator and responder traffic semantics, serializes the extracted telemetry into bincode, and relays the payload to the workstation. It performs no cryptographic proof generation.
• **The Command Node (Verifier, Prover & Interface):** A local workstation executing the RISC Zero STARK proof generation (Host Code), the React WebAssembly verification, the Three.js spatial interface, the server-side FastAPI authorization backend, and the localized LLM via Ollama.
• **WebAuthn Local Domain Exemption:** To fulfill strict WebAuthn Relying Party ID (RPID) registerable-domain requirements and TLS secure context mandates without deploying to the public web, the Command Node utilizes a local DNS mapping (e.g., mapping omniwatch.local to 127.0.0.1 via the OS /etc/hosts file). Both the local reverse proxy TLS certificate (via mkcert) and the FastAPI FIDO2 configurations are explicitly bound to this local domain.

## 4. The Dual-Verification Data Pipeline (Machine + Human)
The system isolates execution logic into verifiable Rust "Guest Code" running within the zkVM, eliminating generative models from the safety-critical detection path.

**Phase 1: Deterministic Ingestion & Edge Serialization**
Zeek (via ICSNPP) passively extracts Modbus TCP metadata. To prevent severe zkVM execution cycle bloat, the Pi 4 serializes the JSON into a flat bincode binary struct before relaying it to the workstation.

**Phase 2: Machine Proof of Execution (RISC Zero zkVM)**
On the Command Node workstation, the Rust Guest Code evaluates the bincode data against the invariant library. The zkVM generates a succinct STARK receipt. To definitively prevent cryptographic replay attacks and cross-session collisions, the Public Journal commits the SHA-256 hash of the serialized input, the Policy Version Identifier, the Deterministic Alert State, and the unique Zeek conn_uid concatenated with a process-boot epoch timestamp.

**Phase 3: Human Proof of Oversight**
• **Pre-Exhibition Credential Enrollment:** The operator registers their WebAuthn/FIDO2 hardware authenticator (e.g., YubiKey or OS Secure Enclave) with the local FastAPI Relying Party database prior to the demonstration.
• **Live Signing:** Upon reviewing the cryptographically verified alert and the LLM-generated context, the operator utilizes their enrolled FIDO2 token to generate an ECDSA signature via the WebCrypto API. This creates an immutable audit trail tying the human's identity to the specific alert context.

**Phase 4: Server-Side Remediation Gating**
The React frontend submits both the succinct STARK receipt (Machine Proof) and the ECDSA signature (Human Proof) as bearer tokens to the localized FastAPI backend. To prevent Time-of-Check to Time-of-Use (TOCTOU) race conditions without blocking the Python GIL, the backend validates the STARK Journal's nonces against a local SQLite "Spent-Receipt Registry" using the `aiosqlite` asynchronous driver and `sqlalchemy[asyncio]`. This utilizes an atomic `INSERT OR IGNORE` operation paired with `PRAGMA journal_mode=WAL`, outright rejecting replayed receipts with zero concurrency flaws. To ensure cryptographic interoperability, the pipeline includes an encoding translation layer converting the browser's native IEEE P1363 raw signatures into ASN.1 DER format. Upon dual-validation, the server executes remediation scripts bounded exclusively to logical network isolation (e.g., firewall updates) and strictly prohibits automated Modbus write commands.

## 5. Design Technology and Human-Computer Interaction (HCI)
The frontend interface utilizes Ecological Interface Design (EID) and Web Workers to process cryptography asynchronously.
• **Asynchronous WASM Pre-Check:** The WebAssembly module (risc0-zkvm) is instantiated entirely within a dedicated background Web Worker. This client-side execution serves strictly as a UX optimization to update the visual state asynchronously.
• **Spatial WebGL DAG:** The verification state is visualized via Three.js as a 3D Directed Acyclic Graph (DAG) explicitly mapping the Cryptographic Trust Chain. Upon successful server-side and Web Worker verification, the DAG solidifies into a verified green state, signaling to the human operator that the alert is mathematically authentic and ready for manual ECDSA signing.
• **Graceful Degradation:** The React architecture integrates strict error boundaries, automatically degrading to a dense, high-performance 2D CSS-grid matrix if the WebGL context fails to initialize (meeting enterprise dashboard expectations).

## 6. LLM RAG Contextualization & Two-Tiered Evaluation
The prototype's operational viability is evaluated across two strictly separated scoreboards to distinguish deterministic cryptographic overhead from language model utility.
• **Scoreboard 1 (Proof-System Metrics):** Benchmarks quantifying the computational cost of the RISC Zero STARK generation, specifically reporting total cycle count, peak RAM consumption during FRI polynomial commitments, output proof size (in KB), end-to-end generation latency (in seconds), Phi-3 RAG token-streaming latency, and the WebAssembly (WASM) bundle size alongside the dashboard's total Time to Interactive (TTI) penalty (accounting for cold-cache compilation).
• **Scoreboard 2 (LLM RAG Contextualization):** The localized 4-bit Phi-3-Mini model operates strictly within a Retrieval-Augmented Generation (RAG) architecture. When an invariant is violated, the LLM queries static, localized engineering manuals (such as SWaT P&ID documentation) to translate the raw Modbus hex violation into human-readable industrial context for the operator. To maintain retrieval fidelity and mathematically stable cosine similarity calculations, the pipeline dictates that while the generation model (Phi-3) utilizes INT4 quantization, the underlying Embedding Model must be strictly locked to FP32 or FP16 precision. The UI enforces strict visible source grounding, displaying the exact document excerpt alongside the LLM summary. Architecturally, the LLM summary carries zero normative weight in the detection verdict. Furthermore, the pipeline explicitly tests adversarial RAG queries and enforces a hard "No Grounding Available" UI state when retrieval confidence falls below the calibrated threshold.

## 7. Commercial and Regulatory Alignment
The architecture functions as cryptographically verified OT triage and authorization support. Due to the 15–32 second total end-to-end operator latency, the system is explicitly designed for post-detection governance, manual remediation approvals, and audit-trail generation, rather than real-time protective response or inline Intrusion Prevention (IPS). This demonstrates a scalable technical mechanism aligning with the monitoring, event collection, and audit-trail integrity objectives of the Saudi National Cybersecurity Authority (NCA) OTCC-1:2022 Subdomain 2-11 (Cybersecurity Event Logs and Monitoring Management). By demanding setup-free cryptographic proof of logical policy evaluation combined with a FIDO2-backed operator cryptographic signature before remediation is authorized, the system addresses the precise operational safety requirements for critical water utility infrastructure within Vision 2030 smart city deployments.
